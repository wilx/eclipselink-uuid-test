# AGENT.md

## Goal

Implement and maintain a focused Spring Boot integration test that reproduces EclipseLink issue eclipse-ee4j/eclipselink#2717 against a real PostgreSQL database provisioned by Testcontainers.

The test must exercise Spring Data JPA repository `save(...)` with EclipseLink `4.0.9`, a PostgreSQL `uuid` column, and a Java `java.util.UUID` entity field that is left unset.

## Technology constraints

Use:

* Spring Boot test infrastructure.
* Spring Data JPA repository.
* EclipseLink `4.0.9` as the JPA provider.
* PostgreSQL via Testcontainers.
* JUnit Jupiter.
* A real PostgreSQL container started by the test itself.
* No external PostgreSQL instance.
* No H2, embedded database, mocks, or fake JDBC layer.

Spring Boot may use Testcontainers’ Spring Boot integration. Prefer `@ServiceConnection` with `PostgreSQLContainer<?>` when available in the project’s Spring Boot version.

## Domain model

Create a JPA entity named `Gizmo`.

The database table must be named `gizmo`.

The table must have exactly these relevant columns:

```sql
id integer primary key generated by default as identity,
gizmo_id uuid
```

The Java entity field mapping must be:

```java
@Entity
@Table(name = "gizmo")
public class Gizmo {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Integer id;

    @Column(name = "gizmo_id")
    private UUID gizmoId;
}
```

Use `java.util.UUID` for the entity field type.

Do not initialize `gizmoId`.

Do not set `gizmoId` to `null` explicitly in the test.

The test should construct the entity as a new object and call the Spring Data JPA repository `save(...)`.

## Repository

Create a Spring Data JPA repository:

```java
public interface GizmoRepository extends JpaRepository<Gizmo, Integer> {
}
```

## Database setup

The test must create the PostgreSQL schema itself.

Acceptable approaches:

1. A test-only SQL initializer, such as `schema.sql`, if it is guaranteed to run against the Testcontainers PostgreSQL database.
2. Programmatic DDL in the test using `JdbcTemplate` before repository save.
3. Flyway or Liquibase only if already present and simple to use.

The resulting table must use PostgreSQL’s real `uuid` type for `gizmo_id`.

Do not rely on Hibernate DDL generation.

Do not use H2 compatibility modes.

## EclipseLink configuration

Ensure Hibernate is not the active JPA provider for this test.

Use EclipseLink `4.0.9`.

If the existing project uses Spring Boot’s default Hibernate configuration, add test-specific configuration so that the test uses EclipseLink.

The configuration may need to provide:

* `EclipseLinkJpaVendorAdapter`
* `LocalContainerEntityManagerFactoryBean`
* `JpaTransactionManager`
* package scanning for the `Gizmo` entity
* the project’s `DataSource`
* Spring Data JPA repository scanning

Keep this configuration test-scoped where possible.

Use Jakarta Persistence imports, not `javax.persistence`.

## Expected behavior

The purpose is to reproduce the EclipseLink/PostgreSQL UUID-null binding defect.

The test should save a `Gizmo` instance with `gizmoId` unset.

The test should assert the behavior that demonstrates the defect. If EclipseLink `4.0.9` still contains the defect, the save/flush operation is expected to fail with a PostgreSQL error similar to:

```text
ERROR: column "gizmo_id" is of type uuid but expression is of type character varying
Hint: You will need to rewrite or cast the expression.
```

Make sure the persistence operation is flushed inside the test so the SQL insert is actually executed during the test method.

Prefer:

```java
assertThatThrownBy(() -> repository.saveAndFlush(new Gizmo()))
    .hasRootCauseInstanceOf(PSQLException.class)
    .hasMessageContaining("gizmo_id")
    .hasMessageContaining("uuid")
    .hasMessageContaining("character varying");
```

If exception wrapping differs, inspect the actual exception chain and assert robustly while still verifying that PostgreSQL rejected the insert because the `uuid` column received a `character varying` expression.

## Test quality requirements

* The test must be self-contained.
* The PostgreSQL database must come from Testcontainers.
* The table must be created by the test or test resources.
* The test must not require a developer’s local PostgreSQL service.
* The test must not require pre-existing schema.
* The test must not pass accidentally because the insert never flushes.
* The test must not set `gizmoId`.
* The test must not explicitly call `setGizmoId(null)`.
* The test must not work around the defect by adding casts, converters, custom binders, or PostgreSQL-specific entity annotations unless such code is clearly isolated in a separate negative/control test.
* Keep the test minimal and focused on EclipseLink issue #2717.
